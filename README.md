# DEFLATE Data Compression Engine

## Overview

This project is a high-performance C++ implementation of the **DEFLATE compression algorithm**, developed as part of the **CSC10004 - Data Structures and Algorithms** course at [VNU-HCMUS](https://hcmus.edu.vn/). 

The engine effectively eliminates data redundancy by combining **LZ77 dictionary compression** and **Huffman Coding**. It is designed to be a fast, lossless, and robust tool for handling various data types, from system logs to large binary archives.

* **Repository:** [nguyenquan1203/ProjectDSA](https://github.com/nguyenquan1203/ProjectDSA)
* **Dataset:** [Download Dataset (Google Drive)](https://drive.google.com/drive/folders/1QzfY9ciPuGCJPWq2nDD5Lfq-7FMsh2tp) — Contains 227 diverse test cases.
* **Core Logic:** Located in `./src` and `./Deflate`.

## Project Report

### 1. Problem 

Storage and bandwidth are finite resources. Motlenecdern data often contains repetitive structural patterns and non-uniform character distributions. The DEFLATE algorithm (widely used in ZIP and GZIP) solves this by combining two different compression paradigms. Our goal was to implement this complex system from scratch, which required overcoming the traditional $O(N \cdot W)$ search botk of the LZ77 algorithm while effectively managing variable-length bit streams through Huffman Coding.

### 2. Core Algorithm & Technical Implementation

The system is built upon three main technical pillars:

#### A. LZ77 Dictionary Compression

LZ77 replaces repeated occurrences of data with references (distance and length) to a single copy of that data existing earlier in the uncompressed stream. To optimize the slow linear search of standard LZ77, we implemented a **Hash Chain** mechanism:
* We compute a hash value for 3-byte sequences.
* A hash table stores the head of a linked list for each hash value, allowing the algorithm to find potential matches in $O(1)$ time and jump through previous occurrences, significantly reducing the search complexity.

#### B. Huffman Entropy Coding

After LZ77 generates a stream of literals and (distance, length) pairs, we apply Huffman Coding to reduce entropy:
* It builds a frequency tree to assign **variable-length bit codes**.
* Frequent symbols receive shorter codes, while rare symbols receive longer ones. This squeezes the data toward its theoretical entropy limit without losing any information.

#### C. Custom Bit Stream I/O

Since C++ only supports byte-level (8-bit) file operations, writing the variable-length codes generated by the Huffman tree required a custom **Bit Stream Module**:
* **Architecture:** Uses a procedural approach with simple `struct` types and global functions for maximum speed and minimal overhead.
* **Mechanism:** Employs a bit buffer and bitwise operators (`<<`, `>>`, `|`, `&`) to pack individual bits into bytes before writing them to the disk, and vice versa during decompression.

### 3. Experimental Analysis

We evaluate the compression efficiency using the following formula:

$$\text{Compression Saving \%)} = \left( 1 - \frac{\text{Size}_{\text{compressed}}}{\text{Size}_{\text{original}}} \right) \times 100\%$$

#### Performance Metrics Summary

Based on our testing across 227 files, the engine achieved a **100% Pass Rate**.

| File Category | Example File | Original Size | Saving (%) | Speed (MB/s) |
| :--- | :--- | :--- | :--- | :--- |
| **Highly Redundant** | `aaa.txt` | 100 KB | **99.91%** | 47.68 |
| **Large Logs** | `openstack_normal2.log` | 40.5 MB | **77.03%** | 10.84 |
| **Source Code** | `encode.c` | 32.5 KB | **35.72%** | 1.94 |
| **Complex Archives** | `mozilla.tar` | 51.2 MB | **50.95%** | 5.38 |

#### Technical Observations

* **System Efficiency:** The DEFLATE system handles various data types exceptionally well. The combination of LZ77 and Huffman maintains steady processing speeds over **10 MB/s** even for large 40MB+ log files.
* **Data Inflation:** For extremely small files (e.g., `< 100 bytes`), we observed "negative saving." This is expected, as the overhead of storing the **Huffman Tree dictionary** inside the file header exceeds the space saved by compression.
* **Memory Trade-off:** For the largest test cases (~50MB), overall RAM usage reached **635 MB**. This indicates a "Speed-over-RAM" design choice, where core data structures and file buffers are kept fully in memory to maximize throughput.

---

### 4. Discussion & Future Work

* **Current Limitations:** The high memory footprint on large files makes it difficult to process GB-sized files on limited hardware environments.
* **Block-based Streaming:** A planned architectural upgrade is to implement **Block-based processing** (e.g., compressing in independent 256KB chunks). This would cap RAM usage to a constant low value regardless of the input size.
* **Adaptive Huffman:** Implementing Dynamic (Adaptive) Huffman trees would allow the engine to compress data in a single pass rather than scanning the file twice (once for frequency counting, once for encoding), significantly reducing I/O latency.
